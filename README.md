# CT-200 Document API

A backend system that parses the CardioTrack CT-200 device manual (PDF)
into a structured, versioned document tree, supports browsing and
version comparison, manages version-pinned selections of document
sections, and generates LLM-powered QA test case ideas with staleness
detection when the source document changes.

Built with FastAPI, SQLAlchemy + SQLite, PyMuPDF/pdfplumber for PDF
parsing, and Hugging Face's Inference API for LLM generation.

---

## Tech Stack

- **FastAPI** + **Pydantic** — API layer and request/response validation
- **SQLAlchemy** + **SQLite** — document tree, versions, selections
- **PyMuPDF (fitz)** — PDF text/font extraction
- **pdfplumber** — table extraction
- **Hugging Face Inference API** — LLM-generated test cases
  *(originally built against Groq; switched after an environment-variable
  precedence issue caused persistent auth failures — see approach.md
  Phase 5 debugging notes for details)*
- **Flat JSON file** (`data/generations.json`) — NoSQL-style store for
  generated test cases
- **pytest** — unit tests

## Setup

### 1. Clone and enter the project

```powershell
git clone https://github.com/RachnA94664/AI_tri9t-ct200-backend.git
cd tri9t-ct200-backend
```

### 2. Create and activate a virtual environment

```powershell
python -m venv venv
venv\Scripts\activate
```

You should see `(venv)` at the start of your prompt once activated.

### 3. Install dependencies

```powershell
pip install -r requirements.txt --break-system-packages
```

### 4. Set up environment variables

Create a `.env` file in the project root (this file is git-ignored and
must be created manually — it is never committed):
HF_API_KEY=hf_your_hugging_face_token_here
To get a Hugging Face token:
1. Sign up / log in at https://huggingface.co
2. Go to https://huggingface.co/settings/tokens
3. Click "Create new token" → select the **"Read"** or **"Inference"**
   preset → copy the token immediately (shown only once)
4. Paste it into `.env` as shown above

**Never commit `.env` or paste tokens into any tracked file** — this
project's own git history includes two incidents of exactly that
(documented transparently in `docs/approach.md`, Phase 5), both caught by
GitHub's push protection before reaching the public repo.

### 5. Place the source PDFs

Ensure both manual versions are present in the `data/` folder:
data/ct200_manual_v1.pdf
data/ct200_manual_v2.pdf

## Running the Server

```powershell
uvicorn app.main:app --reload
```

The API will be available at `http://127.0.0.1:8000`. Interactive API
docs (Swagger UI) are auto-generated by FastAPI at
`http://127.0.0.1:8000/docs`.

**Note on environment variables and `--reload`:** `--reload` watches
Python source files for changes and restarts the server automatically,
but it does **not** reload `.env` values into an already-running process.
If you change `.env` (e.g. update `HF_API_KEY`), you must fully stop
(`Ctrl+C`) and restart uvicorn for the new value to take effect.

**Running the server and sending requests requires two separate
terminal windows** — one running uvicorn (which stays busy and doesn't
accept new commands), and a second one for `curl`/`Invoke-WebRequest`
commands.

## Ingesting a Document Version

Each ingestion parses a PDF into a hierarchical tree and persists it as
a new `DocumentVersion` in SQLite, without touching or deleting any
previously ingested version.

```powershell
python -m app.services.run_ingest data/ct200_manual_v1.pdf v1
python -m app.services.run_ingest data/ct200_manual_v2.pdf v2
```

Verify what's been ingested:

```powershell
python -c "from app.db.base import SessionLocal; from app.models.document import DocumentVersion; db = SessionLocal(); [print(v.id, v.label, v.source_filename) for v in db.query(DocumentVersion).all()]"
```

## API Endpoint Reference

### Browse
| Method | Endpoint | Description |
|--------|----------|-------------|
| GET | `/browse/sections?version=latest` | List top-level sections (version defaults to latest; can pass a label like `v1`) |
| GET | `/browse/nodes/{node_id}` | Full node detail: children, body text, content hash |
| GET | `/browse/search?q=...&version=latest` | Search node headings/body text |
| GET | `/browse/nodes/{node_id}/diff` | Change status + diff summary vs. the adjacent version |

### Selections
| Method | Endpoint | Description |
|--------|----------|-------------|
| POST | `/selections` | Create a named, version-pinned selection from a list of `node_ids` |
| GET | `/selections/{selection_id}` | Retrieve a selection's pinned node+version pairs and resolved text |

### Generation
| Method | Endpoint | Description |
|--------|----------|-------------|
| POST | `/generations/selections/{selection_id}` | Generate 3–5 QA test cases from a selection. Returns existing generation unless `?force=true` |

### Retrieval
| Method | Endpoint | Description |
|--------|----------|-------------|
| GET | `/generations?selection_id=...` | Fetch generations for a selection, each annotated with `staleness` |
| GET | `/generations?node_id=...` | Fetch generations that used a given node as a source |

See `docs/api_examples.md` for full working PowerShell command examples
against each endpoint.

## Triggering the Full v1 → v2 Re-Ingestion + Staleness Flow (End to End)

This is the core workflow the system is built around — proving that a
generation created from v1 text correctly becomes stale once v2 changes
the underlying document.

### Step 1 — Start the server

`````powershell
uvicorn app.main:app --reload
````//Vibe coded fully

### Step 2 — Ingest v1 (in a second terminal)

```powershell
python -m app.services.run_ingest data/ct200_manual_v1.pdf v1
```

### Step 3 — Browse v1 and note a node ID to use

```powershell
Invoke-WebRequest -Uri "http://127.0.0.1:8000/browse/sections?version=v1" -Method GET
```

Pick a `node_id` from the response — e.g. the "Battery Life Under Typical
Use" section, since its content is known to change in v2.

### Step 4 — Create a version-pinned selection

```powershell
Invoke-WebRequest -Uri "http://127.0.0.1:8000/selections" -Method POST -ContentType "application/json" -Body '{"name": "battery-check", "node_ids": [<node_id_from_step_3>]}'
```

Note the returned `id` — this is your `selection_id`.

### Step 5 — Generate test cases from that selection

```powershell
Invoke-WebRequest -Uri "http://127.0.0.1:8000/generations/selections/<selection_id>" -Method POST
```

### Step 6 — Confirm it's fresh

```powershell
Invoke-WebRequest -Uri "http://127.0.0.1:8000/generations?selection_id=<selection_id>" -Method GET
```

Response should show `"staleness":{"is_stale":false,...}`.

### Step 7 — Ingest v2 (the document has now changed)

```powershell
python -m app.services.run_ingest data/ct200_manual_v2.pdf v2
```

This creates a new `DocumentVersion` **without deleting v1 or the
selection/generation created against it**.

### Step 8 — Re-check staleness

```powershell
Invoke-WebRequest -Uri "http://127.0.0.1:8000/generations?selection_id=<selection_id>" -Method GET
```

If the selected node's content changed between v1 and v2 (as it does for
Battery Life: "300 cycles / 15%" → "250 cycles / 10%"), the response now
shows `"staleness":{"is_stale":true,"stale_nodes":[{"node_id":...,"reason":"content_changed"}]}` —
while the original selection and generation still resolve to their exact
original v1 text, unaffected by the re-ingestion.

## Known Limitations

See `docs/approach.md` — Decision Log for full detail. Briefly:
- Staleness detection is a binary content-hash comparison; it cannot
  distinguish a cosmetic wording edit from a safety-critical threshold
  change.
- Version-matching's heading-text fallback assumes unique heading text
  per version, which is not guaranteed in general.
- The generation store (flat JSON file) has no concurrency control.
- Selections spanning multiple document versions are not validated
  against.
`````
